# PC 网页页面功能设计文档

## 1. 文档信息

- 项目名称：演示系统
- 文档类型：PC 网页页面功能设计
- 所属仓库：`yanshi_be`
- 当前状态：初稿
- 创建日期：2026-06-21
- 最后更新：2026-06-21

## 2. 项目范围

当前网站包含以下两个页面：

1. 登录页
2. 首页

本文档用于明确这两个页面的功能目标、页面结构、交互流程及前后端协作边界，作为后续开发与联调依据。

## 3. 整体业务说明

### 3.1 使用端

- `PC` 端网页

### 3.2 用户流程总览

1. 用户访问系统后，默认进入登录页。
2. 用户输入账号和密码进行登录。
3. 登录成功后跳转至首页。
4. 首页用于展示系统主入口信息和基础操作入口。
5. 用户可在首页执行退出登录操作，并返回登录页。

### 3.3 当前默认假设

由于目前仅确认页面数量，以下内容按通用后台管理系统模式设计，后续可继续细化：

- 登录方式为账号密码登录。
- 首页为登录后的系统欢迎页或工作台首页。
- 需要基础登录态校验。
- 首页需具备退出登录能力。

## 4. 页面一：登录页

### 4.1 页面基础信息

- 页面名称：登录页
- 页面路由：`/login`
- 页面类型：表单页
- 使用端：`PC`
- 用户角色：系统用户
- 业务目标：完成用户身份校验，并建立登录态

### 4.2 页面核心流程

1. 用户进入登录页后，可看到系统名称、登录表单和登录按钮。
2. 用户输入账号和密码后，点击登录按钮提交。
3. 前端进行必填校验，校验通过后调用登录接口。
4. 登录成功后，保存登录凭证并跳转至首页。
5. 登录失败时，在页面提示失败原因，用户可重新输入后再次提交。

### 4.3 页面模块拆分

| 模块名称 | 模块说明 | 是否必需 | 备注 |
| --- | --- | --- | --- |
| 品牌展示区 | 展示系统名称、Logo、欢迎文案 | 是 | 可选背景图或视觉装饰 |
| 登录表单区 | 账号输入框、密码输入框、登录按钮 | 是 | 核心交互区 |
| 提示信息区 | 展示登录失败、校验错误等提示 | 是 | 与表单联动 |
| 页脚信息区 | 展示版权或备案信息 | 否 | 可后续补充 |

### 4.4 字段设计

| 字段名称 | 字段编码 | 类型 | 是否必填 | 展示位置 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 账号 | `username` | string | 是 | 登录表单 | 支持用户名或手机号，待最终确认 |
| 密码 | `password` | string | 是 | 登录表单 | 输入时需掩码显示 |

### 4.5 交互与状态设计

- 默认状态下展示空表单。
- 点击登录时，如字段为空，展示必填校验提示。
- 登录提交过程中，登录按钮进入加载中状态，避免重复提交。
- 登录失败后，保留账号输入内容，密码框清空或保留由产品决定，当前建议清空。
- 如用户已处于有效登录状态，再访问登录页时可直接跳转首页。
- 支持按回车键提交登录。

### 4.6 权限设计

| 权限点 | 权限说明 | 影响范围 |
| --- | --- | --- |
| public_login | 无需登录可访问登录页 | 是否允许进入登录页 |
| auth_login_submit | 提交登录请求 | 是否允许发起登录 |

### 4.7 接口设计草案

| 接口名称 | 方法 | 路径 | 用途 | 备注 |
| --- | --- | --- | --- | --- |
| 用户登录 | `POST` | `/api/auth/login` | 校验账号密码并返回登录凭证 | 返回 token 或 session |
| 获取当前用户信息 | `GET` | `/api/auth/me` | 获取当前登录用户基础信息 | 用于登录态恢复 |
| 退出登录 | `POST` | `/api/auth/logout` | 注销当前登录态 | 可在首页调用 |

### 4.8 数据校验规则

- `username` 不能为空。
- `password` 不能为空。
- 密码输入框需限制最大长度，默认建议不低于 20 位。
- 前端做基础非空校验，后端负责账号密码真实性校验。
- 登录失败提示语避免暴露过多账号安全信息。

### 4.9 验收标准

1. 用户可正常进入登录页。
2. 账号和密码为空时，页面可提示必填错误。
3. 正确账号密码可成功登录并跳转首页。
4. 错误账号密码无法登录，并展示失败提示。
5. 登录请求提交过程中不可重复连续点击。

## 5. 页面二：首页

### 5.1 页面基础信息

- 页面名称：首页
- 页面路由：`/`
- 页面类型：展示页 / 工作台页
- 使用端：`PC`
- 用户角色：已登录系统用户
- 业务目标：展示系统主页面内容，并提供基础导航与退出入口

### 5.2 页面核心流程

1. 用户登录成功后进入首页。
2. 页面初始化时校验登录状态，并获取当前用户信息。
3. 首页展示欢迎信息、系统概览信息或主要功能入口。
4. 用户可通过首页继续进入后续业务模块。
5. 用户点击退出登录后，清理登录态并跳转回登录页。

### 5.3 页面模块拆分

| 模块名称 | 模块说明 | 是否必需 | 备注 |
| --- | --- | --- | --- |
| 顶部栏 | 展示系统名称、用户信息、退出按钮 | 是 | 首页通用区域 |
| 欢迎区 | 展示欢迎文案、当前用户名称 | 是 | 增强进入感 |
| 内容展示区 | 展示首页核心信息或快捷入口 | 是 | 后续可扩展 |
| 快捷操作区 | 展示常用操作入口 | 否 | 视业务需要补充 |

### 5.4 字段设计

| 字段名称 | 字段编码 | 类型 | 是否必填 | 展示位置 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 用户名称 | `nickname` | string | 否 | 顶部栏 / 欢迎区 | 来自用户信息接口 |
| 用户角色名称 | `roleName` | string | 否 | 顶部栏 / 欢迎区 | 可选展示 |
| 欢迎语 | `welcomeText` | string | 否 | 欢迎区 | 前端静态配置也可 |

### 5.5 交互与状态设计

- 用户未登录访问首页时，应自动跳转到登录页。
- 页面首次加载时展示加载状态，等待用户信息返回。
- 获取数据成功后展示首页内容。
- 如果首页暂无业务数据，展示默认欢迎内容和空状态提示。
- 点击退出登录后弹出确认或直接退出，当前建议直接退出。

### 5.6 权限设计

| 权限点 | 权限说明 | 影响范围 |
| --- | --- | --- |
| home_view | 首页访问权限 | 是否允许进入首页 |
| user_info_view | 用户信息查看权限 | 是否展示当前用户信息 |
| logout_submit | 退出登录权限 | 是否可执行退出 |

### 5.7 接口设计草案

| 接口名称 | 方法 | 路径 | 用途 | 备注 |
| --- | --- | --- | --- | --- |
| 获取当前用户信息 | `GET` | `/api/auth/me` | 获取首页展示所需用户信息 | 可与登录态校验复用 |
| 退出登录 | `POST` | `/api/auth/logout` | 清理当前登录态 | 退出后跳转登录页 |
| 首页概览数据 | `GET` | `/api/home/overview` | 获取首页展示内容 | 如当前无业务数据，可暂不实现 |

### 5.8 数据校验规则

- 首页接口需校验用户登录态。
- 首页返回字段允许部分为空，但接口结构应稳定。
- 退出登录后前端必须清除本地登录凭证。

### 5.9 验收标准

1. 登录成功后可自动进入首页。
2. 未登录用户访问首页会跳转登录页。
3. 首页可正常展示当前用户基础信息。
4. 用户可从首页退出登录并返回登录页。
5. 首页在无业务数据时仍可正常展示基础欢迎内容。

## 6. 前后端协作建议

### 6.1 前端建议

- 统一处理路由鉴权。
- 统一封装 token 存储与请求头注入。
- 统一处理接口异常提示和登录失效跳转。

### 6.2 后端建议

- 提供标准登录接口与用户信息接口。
- 定义统一响应结构，便于前端处理。
- 对未登录或 token 失效场景返回明确状态码。

### 6.3 响应结构建议

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 7. 当前待确认项

以下内容建议后续继续明确，以便把文档升级到开发定稿版：

1. 首页到底是纯欢迎页，还是带业务数据看板。
2. 登录账号类型是用户名、手机号还是邮箱。
3. 是否需要验证码。
4. 是否需要记住登录状态。
5. 首页是否需要左侧菜单、顶部导航或快捷入口卡片。
6. 是否存在不同角色看到不同首页内容的需求。
